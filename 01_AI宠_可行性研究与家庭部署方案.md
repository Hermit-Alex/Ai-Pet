# AI宠：可行性研究与家庭部署方案

生成日期：2026-06-23  
阶段定位：第一个计划文件 / 家庭 MVP 可行性验证

## 1. 执行摘要

“AI宠”的核心不是简单做一个宠物监控机器人，而是给猫咪、狗狗这类家庭宠物建立一个可交互的“家庭身份”：它能被家人通过社交账号联系，能基于摄像头、喂食器、声音、健康日程和家庭记忆生成符合宠物人设的回应，也能在需要时提醒驱虫、疫苗、洗澡、喂食、异常叫声、长期不活动等事件。

结论：这个项目作为家庭 MVP 非常可行，作为商业化解决方案有机会，但必须从第一天避开“非官方控制个人微信/QQ 客户端”这个高风险路线。建议 MVP 先用 QQ 官方机器人、企业微信/公众号/小程序、Telegram 等开放接口跑通消息闭环；个人微信或个人 QQ 客户端 GUI 自动化只适合作为短期私有实验，不适合作为产品底座。

建议的第一版目标：

1. 老婆给“猫咪账号”发消息或语音吐槽，AI宠能以猫咪口吻回复。
2. Agent 能读取猫咪实时状态：在睡觉、吃饭、喝水、上厕所、活动、叫唤、可能异常。
3. Agent 能查询家庭记忆：驱虫、洗澡、疫苗、出门、遛猫、绝育、疾病、偏好、家庭梗。
4. Agent 能把重要事件沉淀进 RAG 和结构化日历。
5. Agent 的对外动作有权限控制：日常闲聊可自动，健康异常和设备控制要通知人确认。

推荐路线：

- 家庭 MVP：当前 Windows 设备开发，Docker/WSL + Home Assistant + MQTT + OpenClaw + RAG。
- 稳定家庭部署：Mac mini M4/M4 Pro 作为主机，树莓派 5 作为轻量设备网关或备用节点。
- 产品化版本：硬件包 + 本地网关 + 官方社交渠道插件 + 宠物记忆系统 + 家庭隐私与安全策略。

### 1.1 可行性总评

| 维度 | 评分 | 判断 |
| --- | --- | --- |
| 家庭 MVP 技术可行性 | 8/10 | 用现有开源组件和少量自研 Bridge 可以跑通，难点主要在设备接入和权限控制 |
| 情绪价值 / 用户需求 | 9/10 | “宠物成为可联系的家庭成员”很有辨识度，适合从真实家庭故事切入 |
| 社交平台稳定性 | 5/10 | QQ 官方机器人较稳；个人微信/QQ 自动化不适合做产品底座 |
| 宠物状态识别准确性 | 6/10 | 吃饭、活动、叫声等粗粒度可做；“翻译猫语”不能过度承诺 |
| 家庭隐私与安全 | 6/10 | 本地优先可降低风险，但音视频采集、Agent 工具权限必须严控 |
| 轻量硬件部署 | 7/10 | Mac mini M4 很合适；树莓派适合网关，不适合单独跑完整多模态 Agent |
| 商业化可行性 | 7/10 | 可以做，但必须产品化官方通道、标准硬件、隐私方案和安装流程 |

## 2. 项目定义

### 2.1 产品愿景

AI宠是一个“宠物数字家庭成员系统”。它不替代真实宠物，也不声称真正理解宠物语言，而是把宠物的可观测状态、家庭共同记忆和拟人化互动组织起来，让家人可以像联系家庭成员一样联系宠物。

对家庭用户的价值：

- 情绪价值：把“给猫咪打电话吐槽”变成真的可互动体验。
- 关系记录：宠物的生日、疫苗、洗澡、驱虫、旅行、疾病、趣事不再散落在聊天记录里。
- 照护提醒：喂食、喝水、猫砂盆、异常叫声、长时间不动可被及时发现。
- 陪伴增强：宠物不在身边时，家人仍能看到宠物状态并收到带有人设的回应。

对未来商业化的价值：

- 可以从猫扩展到狗。
- 可以从家庭玩具扩展到家庭照护、宠物健康管理、宠物店/寄养场景。
- 可以做成“本地优先、隐私优先”的智能宠物中枢，而不是又一个云摄像头 App。

### 2.2 MVP 边界

第一版不要追求“真能直接接听微信语音电话”。更现实的 MVP 是：

- 通过社交消息、语音消息、网页通话页或小程序实现“像在打电话”的体验。
- 通过 TTS 让猫咪回复语音。
- 通过 STT 把家人的语音转成文本。
- 通过摄像头和传感器给 Agent 提供实时上下文。

真正的微信/QQ 语音或视频通话自动接听，在官方机器人 API 里通常不是稳定开放能力。若靠 GUI 自动化点击客户端，短期能玩，长期会遇到封号、协议变化、稳定性和合规风险。

## 3. 外部技术与平台可行性

### 3.1 OpenClaw

OpenClaw 官方定位是自托管 Gateway：把聊天应用、插件、Web 控制界面、节点和 Agent 连接起来，运行在用户自己的机器或服务器上。官方文档说明它支持多渠道 Gateway、会话、媒体、节点和工具能力，并建议使用 Node 24 或 Node 22 LTS。

对 AI宠的意义：

- 适合作为“家庭 Agent 网关”：接收消息、触发技能、维护记忆、调用工具。
- 适合做本地部署：家庭数据尽量留在本机或家庭网络。
- 适合做多 Agent：一个宠物人格 Agent、一个设备状态 Agent、一个健康日程 Agent、一个安全审计 Agent。

主要风险：

- Agent 拥有工具执行能力，安全面大。
- 不建议把一个 Gateway 暴露给多个互不信任的人。
- 需要限制谁能发消息、能调用哪些工具、能访问哪些设备和文件。

必须采用的硬化策略：

- Gateway 先绑定本机回环或内网，不直接暴露公网。
- 只允许家人账号触发 Agent。
- 设备控制和文件写入默认最小权限。
- 健康异常、摄像头外发、喂食器控制、门锁/安防相关动作必须二次确认。
- 定期运行 `openclaw security audit`。

### 3.2 Hermes

你提到的 “hermes+openclaw” 里，Hermes 可能有两种理解：

- 如果指 HERMES 多模态边缘感知框架：它适合做多传感器采集、同步、边缘推理和事件流处理。
- 如果指 Hermes 系列模型：它适合作为本地或私有化大模型的一种候选。

建议：

- MVP 不要一开始就把 Hermes 作为强依赖，先用 Home Assistant + MQTT 把设备状态统一起来。
- 当摄像头、麦克风、穿戴设备、猫砂盆等数据源变多，再把 Hermes 类框架引入为“实时多模态感知层”。
- 模型层保持可替换：OpenAI/Claude/DeepSeek/本地 Ollama/Hermes 模型都可以是同一个 Agent 接口后的实现。

### 3.3 QQ / 微信通道

#### QQ 推荐路线

优先使用 QQ 官方机器人平台。QQ 机器人官方文档说明其提供开放接口、API、webhook 事件，并覆盖 QQ频道、群、单聊等场景。个人开发者也可以入驻，但企业主体和个人主体在后续能力申请上可能不同。

适合 MVP 的原因：

- 官方机器人身份比模拟个人 QQ 更稳定。
- 支持沙箱环境，便于先在家庭小范围测试。
- 可做群聊、单聊、指令面板。
- 未来产品化时更容易解释合规边界。

限制：

- 它是“机器人”，不一定能完全像一个普通宠物个人号。
- 需要审核、配置白名单和消息回调。
- 语音/视频通话能力需要以官方实际开放能力为准，不能假设可用。

#### 微信推荐路线

微信个人号自动化不要作为产品底座。更稳妥的路径是：

- 家庭私有测试：可先用企业微信群机器人、公众号测试号、小程序、Web 通话页来做闭环。
- 产品化：优先做小程序 + 公众号/服务号 + 企业微信/微信客服等官方能力组合。
- 如果未来腾讯官方提供稳定的 WeChat/OpenClaw Agent 接入，再评估替换。

不建议路线：

- 直接操控个人微信客户端。
- 使用非官方逆向协议。
- 批量注册宠物微信号用于自动回复。

这些路线在家庭玩具阶段也许能临时跑，但对商业化很危险。

## 4. 需求与场景拆解

### 4.1 核心场景：给猫咪打电话吐槽

第一阶段“电话”的实现建议：

1. 老婆给猫咪 QQ 机器人或微信小程序发语音/文字。
2. STT 把语音转文字。
3. Agent 检索猫咪当前状态与家庭记忆。
4. Agent 以猫咪人设生成回应。
5. TTS 生成语音，发回社交通道。
6. 可选：家中音箱播放“猫咪替你回话”。

示例体验：

- 老婆：“你爸今天又乱放袜子！”
- AI宠读取状态：“猫刚吃完饭，现在趴在窗边。”
- 猫咪回复：“本喵已记录。建议把袜子放到他键盘上，形成正向教育。顺便说一句，我刚吃饱，不接受你爸以零食贿赂撤诉。”

第二阶段“真通话感”的实现建议：

- 做一个家庭内网 WebRTC 页面：点击“呼叫猫咪”，打开语音对话。
- 页面里展示摄像头快照/状态卡。
- 后端实时 STT/TTS。
- 小程序或 QQ 机器人按钮跳转到该页面。

第三阶段才考虑：

- 官方平台若开放实时音视频机器人能力，再接入。
- 不建议依赖 GUI 自动接听微信/QQ 通话。

### 4.2 宠物状态感知

状态源分为三类：

- 设备事件：喂食器出粮、饮水机水量、猫砂盆使用、项圈定位、电量。
- 视觉事件：摄像头检测到宠物、睡觉、活动、进食、上厕所、异常姿态。
- 声音事件：喵叫、持续叫声、疑似痛苦叫声、狗叫、异常撞击声。

MVP 先做“可解释粗分类”：

- `sleeping`
- `eating`
- `drinking`
- `litter_box`
- `active`
- `not_seen`
- `meowing`
- `possible_distress`

不要在第一版宣称“准确翻译猫语”。猫叫声识别可以先做事件检测和异常提醒，而不是语义翻译。

### 4.3 家庭记忆与 RAG

记忆分两层：

- 结构化事件：日期、类型、参与人、宠物、地点、设备、备注、下次提醒时间。
- 非结构化记忆：家庭故事、猫咪偏好、口头禅、照片说明、旅行记录、兽医建议。

建议事件类型：

- 健康：疫苗、驱虫、体检、绝育、疾病、用药、体重。
- 护理：洗澡、剪指甲、梳毛、清洁耳朵。
- 日常：喂食、饮水、猫砂盆、睡眠、活动。
- 外出：出门、遛猫、寄养、宠物医院、旅行。
- 情绪：家人吐槽、猫咪趣事、家庭梗。

RAG 只负责“帮 Agent 想起相关记忆”，权威健康逻辑仍然要用结构化规则和人工确认。

## 5. 总体架构

推荐架构：

```text
家人社交入口
  ├─ QQ 官方机器人 / 沙箱单聊 / 家庭群
  ├─ 微信小程序 / 公众号 / 企业微信机器人
  └─ WebRTC 家庭通话页

OpenClaw Gateway
  ├─ 宠物人格 Agent
  ├─ 设备状态 Agent
  ├─ 健康日程 Agent
  └─ 安全审核 / 人工确认策略

AI宠 Bridge 服务
  ├─ Home Assistant API
  ├─ MQTT 订阅
  ├─ 摄像头快照 / RTSP 抽帧
  ├─ 声音事件识别
  ├─ RAG 检索
  └─ TTS/STT 服务

家庭设备层
  ├─ ONVIF/RTSP 摄像头
  ├─ 宠物喂食器
  ├─ 饮水机
  ├─ 猫砂盆
  ├─ 宠物随身摄像头/项圈
  ├─ 麦克风/智能音箱
  └─ 安防传感器

数据层
  ├─ SQLite/PostgreSQL：结构化事件
  ├─ Qdrant/Chroma/LanceDB：向量记忆
  ├─ 对象存储/本地 NAS：图片、音频、短视频
  └─ 审计日志：谁触发、用了什么数据、做了什么动作
```

### 5.1 Home Assistant

Home Assistant 适合做智能家居设备中枢。它支持 ONVIF 摄像头、MQTT、Ollama、OpenAI Conversation 等集成。对 AI宠而言，Home Assistant 不负责“人格”，只负责把设备状态标准化。

建议把 Home Assistant 作为设备层唯一入口：

- 设备尽量先进 Home Assistant。
- AI宠 Bridge 只读 Home Assistant 实体状态和事件。
- 需要控制设备时，通过 Home Assistant 服务调用，并加权限和确认。

### 5.2 MQTT

MQTT 适合传递轻量设备事件：

- `aipet/cat/state`
- `aipet/cat/feeding`
- `aipet/cat/litter`
- `aipet/cat/audio`
- `aipet/cat/camera/front_door`
- `aipet/system/alert`

建议使用 Mosquitto Broker，账号分级：

- 设备只允许发布自己的 topic。
- Bridge 可订阅设备 topic。
- Agent 不直接拿 MQTT 管理员账号。

### 5.3 摄像头

建议购买或优先使用支持以下能力的摄像头：

- RTSP 或 ONVIF。
- H.264 子码流。
- 本地局域网访问。
- 可关闭云端上传或至少可限制外网。
- 夜视。
- 麦克风可选，但卧室等隐私区域慎用。

推荐部署点：

- 客厅广角摄像头：识别活动、睡觉、玩耍。
- 喂食区摄像头：识别吃饭。
- 饮水区摄像头：识别饮水。
- 猫砂盆附近摄像头：只拍入口/轮廓，不拍过度隐私细节。
- 门口/阳台：安全提醒。

宠物随身摄像头建议谨慎：

- 重量要低，不能影响猫行动。
- 发热、电池、佩戴舒适度比画质重要。
- 第一版可以不用，先用固定摄像头完成体验。

### 5.4 声音识别

MVP 可用 YAMNet 或 TensorFlow Lite 音频分类做“喵叫事件检测”。TensorFlow 官方 YAMNet 教程说明该模型能预测 AudioSet 的 521 类音频事件，并使用 16kHz mono wav 输入。

建议第一版只输出：

- 是否有猫叫。
- 叫声频率是否异常。
- 是否持续超过阈值。
- 是否需要家人看一眼。

不要输出：

- “猫咪现在说它想你了”这类确定性翻译。
- 医疗诊断。

## 6. 家庭硬件建议

### 6.1 当前开发硬件

当前这台 Windows 设备可以作为开发主机：

- 跑 OpenClaw Gateway。
- 跑 Docker Desktop 或 WSL2。
- 跑 Home Assistant 容器或 HAOS 虚拟机。
- 跑 RAG、Bridge 服务、轻量推理。
- 摄像头/VLM 推理可以先抽帧调用云模型，降低本地压力。

### 6.2 稳定部署：Mac mini M4 / M4 Pro

Mac mini M4 很适合家庭常驻 Agent：

- 体积小、功耗低、性能强。
- Apple 官方规格显示 M4 版有 10 核 CPU、10 核 GPU、16 核 Neural Engine、120GB/s 内存带宽；M4 Pro 有更高 CPU/GPU 和 273GB/s 内存带宽。
- M4 版 16GB 可以跑 MVP；建议选 24GB 内存。
- 如果要本地多模态模型、多个摄像头分析、长上下文 RAG，建议 M4 Pro 48GB。

推荐配置：

- 起步：Mac mini M4，24GB RAM，512GB/1TB SSD。
- 更稳：Mac mini M4 Pro，48GB RAM，1TB SSD，10GbE 可选。
- 外接：USB/Thunderbolt SSD 或 NAS 存储媒体文件。

### 6.3 轻量部署：Raspberry Pi 5

树莓派 5 可以作为设备网关，但不建议单独承担完整多模态 Agent：

- 官方规格显示 Raspberry Pi 5 使用 Broadcom BCM2712 2.4GHz 四核 Cortex-A76，RAM 最高到 16GB。
- 适合跑 Home Assistant、Mosquitto、简单 Bridge、设备采集。
- 加 SSD 和主动散热，不建议长期依赖 microSD。
- 本地 LLM 和 VLM 能力有限，建议把大模型调用放到 Mac mini 或云端。

如果要在树莓派上做视觉推理，可考虑 Raspberry Pi AI HAT+ / AI Kit。官方 AI Kit 使用 Hailo-8L，13 TOPS，适合兼容模型的边缘推理，但模型适配成本高，不适合作为第一阶段必需品。

### 6.4 设备清单

最低可行版：

- 1 台开发主机：当前 Windows 设备。
- 1-2 个支持 RTSP/ONVIF 的室内摄像头。
- 1 个宠物喂食器，最好能接 Home Assistant、MQTT 或有本地 API。
- 1 个 USB 麦克风或 Home Assistant Voice 类设备。
- 1 个智能音箱或小音箱，用于 TTS 播放。
- 1 个稳定路由器，支持给 IoT 设备单独 Wi-Fi/VLAN 更好。

增强版：

- 猫砂盆传感器或智能猫砂盆。
- 饮水机水量/使用传感器。
- 门磁、人体/宠物活动传感器。
- NAS 或外接硬盘。
- UPS，防止断电损坏数据库。

产品化探索版：

- 标准化摄像头套装。
- 标准化麦克风/音箱节点。
- 轻量边缘网关。
- 可选宠物项圈/随身摄像头，但需严格验证舒适性和安全。

## 7. 家庭部署步骤

### 阶段 0：准备

1. 建一个专用邮箱和开发者账号，用于 QQ 机器人、云模型、域名、证书。
2. 建一个家庭内网域名或 Tailscale tailnet，避免直接公网暴露。
3. 给 IoT 设备单独 Wi-Fi 或 VLAN。
4. 盘点现有设备：摄像头品牌、是否支持 RTSP/ONVIF、喂食器是否有 API、是否能进 Home Assistant。
5. 明确隐私边界：卧室、卫生间等区域不接入或默认打码。

### 阶段 1：设备中枢

1. 在当前 Windows 设备上安装 Docker Desktop 或 WSL2。
2. 部署 Home Assistant。
3. 部署 Mosquitto MQTT。
4. 添加摄像头：优先 ONVIF；无法 ONVIF 时用 RTSP。
5. 添加喂食器、饮水机、灯、音箱等已有设备。
6. 在 Home Assistant 中建立实体命名规范：
   - `camera.living_room_pet`
   - `camera.feeder`
   - `sensor.feeder_last_feed`
   - `binary_sensor.pet_motion`
   - `sensor.cat_last_seen`

验收标准：

- Home Assistant 能看到至少一个摄像头快照。
- 能读取喂食器或手工模拟喂食事件。
- MQTT 能收到一条 `aipet/test` 消息。

### 阶段 2：OpenClaw Gateway

1. 安装 Node 24 或 Node 22 LTS。
2. 安装 OpenClaw：
   ```bash
   npm install -g openclaw@latest
   openclaw onboard --install-daemon
   openclaw dashboard
   ```
3. Dashboard 默认优先只开本机访问。
4. 配置安全基线：
   - `gateway.bind = loopback`
   - token 使用长随机字符串。
   - 只允许家人账号触发。
   - 默认关闭 shell/文件/自动化高危工具。
5. 运行安全检查：
   ```bash
   openclaw security audit
   ```

验收标准：

- 本机能打开 OpenClaw Dashboard。
- 只有允许的账号能触发 Agent。
- Agent 能调用一个只读测试工具，例如读取 `cat_status.json`。

### 阶段 3：AI宠 Bridge

Bridge 是自研的最小服务，负责把设备世界翻译成 Agent 能理解的工具。

建议接口：

- `get_pet_current_state(pet_id)`
- `get_recent_events(pet_id, hours)`
- `get_camera_snapshot(camera_id)`
- `add_pet_memory(pet_id, text, tags)`
- `add_health_event(pet_id, event_type, date, next_due_date)`
- `request_human_confirmation(action, reason)`
- `publish_pet_voice(text)`

第一版技术选型：

- Node.js 或 Python FastAPI。
- SQLite 存结构化事件。
- Chroma 或 Qdrant 存向量记忆。
- 本地文件夹存图片/音频。
- MQTT 订阅设备事件。

验收标准：

- Bridge 能从 Home Assistant 读取一个实体状态。
- Bridge 能把“刚刚喂食”写入数据库。
- Agent 能查询“猫现在在哪里/上次吃饭什么时候”。

### 阶段 4：社交通道

优先路径：QQ 官方机器人。

1. 注册 QQ 开放平台。
2. 创建机器人。
3. 配置沙箱群、沙箱单聊或频道。
4. 配置 webhook / websocket 回调。
5. 将消息转发到 OpenClaw 或 AI宠 Bridge。
6. 只允许你和老婆两个账号测试。

微信侧第一版：

- 使用企业微信群机器人做家庭群通知。
- 或做一个微信小程序/网页通话页。
- 暂不做个人微信客户端自动化。

验收标准：

- 老婆发一句“猫咪你在吗”，能收到猫咪口吻回复。
- 回复里能带上真实状态，例如“我刚刚吃过饭，现在在客厅”。

### 阶段 5：RAG 与健康日程

1. 建宠物档案：
   - 名字、生日、品种、性别、绝育情况、体重。
   - 性格、人设、常用语气。
   - 禁忌：不编造医疗诊断、不替主人做高风险决定。
2. 录入历史事件：
   - 最近一次驱虫。
   - 最近一次疫苗。
   - 最近一次洗澡。
   - 常去宠物医院。
   - 出去玩、遛猫、寄养记录。
3. 建提醒规则：
   - 驱虫周期。
   - 疫苗周期。
   - 体重记录周期。
   - 洗澡/剪指甲周期。
4. 记忆写入策略：
   - 家人明确说“记一下”才写长期记忆。
   - 设备事件默认只保留摘要。
   - 摄像头原始视频不长期保存，除非出现异常。

验收标准：

- 问“上次驱虫是什么时候”，能准确回答。
- 到期前能发提醒。
- 闲聊时能自然引用家庭梗，但不暴露隐私。

### 阶段 6：声音与视觉

声音：

1. 接入麦克风。
2. 每 1-5 秒切片，跑 YAMNet/TFLite。
3. 检测猫叫事件。
4. 超阈值时写入 `meowing` 事件。
5. 持续异常时通知家人。

视觉：

1. 摄像头每 5-30 秒抽帧。
2. 先用轻量目标检测判断是否看到宠物。
3. 对关键帧用 VLM 生成描述。
4. 把描述摘要进事件流，不把所有图片都进 RAG。

验收标准：

- 猫叫时家庭群能收到“听到猫叫，持续 20 秒”。
- 摄像头看到猫在喂食区时，状态更新为 `eating`。

## 8. 安全、隐私与合规

家庭内部也要按产品化标准做隐私设计，因为这个项目天然会采集家庭音视频。

必须原则：

- 明示同意：家庭成员都知道哪些设备在采集。
- 最小采集：不需要的视频不存，不需要的声音不录。
- 本地优先：原始视频、音频尽量留在家庭网络。
- 分级授权：闲聊、健康、设备控制、外发媒体分别授权。
- 可删除：任何家庭成员可以要求删除自己的对话或相关媒体。
- 审计：记录 Agent 何时读取了摄像头、发出了什么消息、调用了什么工具。

高风险动作必须人工确认：

- 给外部联系人发送图片/视频/音频。
- 控制喂食器额外出粮。
- 调用安防摄像头、门锁、窗帘。
- 写入健康结论或用药建议。
- 修改长期记忆。

社交平台风险：

- 不使用非官方协议批量操控账号。
- 不刷屏、不营销、不陌生人群发。
- 不把宠物账号包装成人类账号。
- 产品化时优先使用官方机器人、小程序、公众号、企业微信、QQ 机器人等开放能力。

## 9. 产品化可行性

### 9.1 目标用户

第一批用户：

- 愿意折腾智能家居的养猫/养狗家庭。
- 经常出差、旅行，想远程陪伴宠物的人。
- 已经有宠物摄像头、喂食器，但觉得多个 App 割裂的人。
- 喜欢家庭情绪价值和拟人化互动的人。

第二批用户：

- 宠物店、猫咖、寄养机构。
- 多宠家庭。
- 老人/孩子和宠物陪伴场景。

### 9.2 差异化

市面上常见产品是：

- 宠物摄像头。
- 自动喂食器。
- 智能猫砂盆。
- 宠物健康记录 App。

AI宠的差异化是：

- 把设备状态、家庭记忆和社交互动合在一起。
- 宠物不是“被监控对象”，而是“可联系的家庭成员”。
- 以本地优先和家庭隐私为卖点。
- 可跨设备品牌整合。

### 9.3 商业化路径

路径 A：家庭开源/极客套件

- 提供 Docker Compose、Home Assistant 配置、OpenClaw 技能、QQ/微信通道模板。
- 用户自己准备硬件。
- 适合验证需求和积累社区。

路径 B：软硬件一体盒子

- 一个 AI宠网关。
- 配套摄像头/麦克风/音箱。
- 手机端或小程序配置。
- 按家庭一次性购买 + 可选云模型订阅。

路径 C：宠物服务机构版

- 多宠档案。
- 多摄像头。
- 异常提醒。
- 家长通知。
- 但必须更严格处理隐私授权。

### 9.4 最大不确定性

1. 社交平台开放能力：个人微信/QQ 不能作为稳定商业化接口。
2. 硬件兼容：宠物设备品牌多，API 封闭。
3. 模型成本：视觉和语音长期运行成本可能高。
4. 准确率：宠物状态识别不能过度承诺。
5. 隐私信任：家里有摄像头和麦克风，用户会敏感。
6. 宠物安全：任何佩戴设备都要考虑重量、发热、误吞、惊吓。

## 10. 里程碑

### M0：一周内

- 完成设备盘点。
- 跑起 Home Assistant。
- 接入至少一个摄像头。
- 建立宠物档案和 SQLite 事件表。
- 完成“手动输入状态 -> 猫咪回复”的最小 Demo。

### M1：两到三周

- 跑起 OpenClaw。
- 接入 QQ 官方机器人沙箱。
- Agent 能读 Home Assistant 状态。
- Agent 能查询 RAG。
- 跑通“老婆发消息吐槽 -> 猫咪带状态回复”。

### M2：一个月

- 接入喂食器/饮水/猫砂盆中至少一个真实设备。
- 接入声音检测。
- 加 TTS 语音回复。
- 加健康日程提醒。
- 做安全审计和人工确认。

### M3：两到三个月

- 做 WebRTC “呼叫猫咪”页面。
- 多摄像头状态融合。
- 本地模型与云模型混合策略。
- 打包 Docker Compose。
- 写第一版产品说明和安装手册。

### M4：产品化探索

- 选择标准硬件包。
- 做非技术用户安装流程。
- 做隐私白皮书。
- 做官方社交通道审核。
- 找 3-5 个朋友家庭试用。

## 11. 推荐技术栈

开发语言：

- Bridge：Python FastAPI 或 Node.js。
- Agent 工具：OpenClaw skills / HTTP tools。
- 前端：Next.js 或轻量 Vite。

数据：

- SQLite：MVP 足够。
- PostgreSQL：多家庭、多宠产品化。
- Qdrant/Chroma/LanceDB：向量检索。
- 本地文件/NAS：媒体。

智能家居：

- Home Assistant。
- MQTT / Mosquitto。
- ONVIF / RTSP / FFmpeg。

模型：

- 对话：云模型优先，后续本地小模型兜底。
- 语音转文字：云 STT 或本地 Whisper。
- 文字转语音：云 TTS 或本地 Piper/系统 TTS。
- 音频事件：YAMNet/TensorFlow Lite。
- 视觉：先云 VLM 抽帧，后续 Frigate/本地检测模型。

部署：

- 当前 Windows：Docker Desktop/WSL2。
- Mac mini：Docker/Colima/OrbStack + Homebrew。
- Raspberry Pi：Home Assistant OS 或 Raspberry Pi OS + Docker。

## 13. 风险矩阵

| 风险 | 概率 | 影响 | 应对 |
| --- | --- | --- | --- |
| 个人微信/QQ 自动化封号或失效 | 高 | 高 | MVP 使用官方机器人/小程序/企业微信；GUI 自动化仅私有实验 |
| 摄像头/喂食器无法本地接入 | 中 | 中 | 优先购买 ONVIF/RTSP/MQTT/Home Assistant 兼容设备 |
| Agent 误操作设备 | 中 | 高 | 最小权限 + 人工确认 + 审计日志 |
| 家庭隐私泄露 | 中 | 高 | 本地优先、媒体最小保存、外发确认、独立 IoT 网络 |
| 猫叫声误判 | 高 | 中 | 只做异常提醒，不做确定翻译 |
| 树莓派性能不足 | 中 | 中 | 树莓派只做网关；Mac mini 或云端跑模型 |
| 多设备长期维护复杂 | 高 | 中 | 标准化硬件清单和安装脚本 |
| 商业化审核不通过 | 中 | 高 | 提前按官方开放平台能力设计 |

## 14. 我的建议

这个想法值得做，而且最好从“家庭小魔法”开始，而不是一开始做成严肃医疗或宠物健康平台。它最打动人的地方是：猫咪不是数据看板，而是家庭关系里的一个角色。技术方案要服务这个体验。

我建议第一版只追求一个漂亮闭环：

> 老婆在外面给猫咪发消息吐槽你，猫咪根据自己当前状态、家庭记忆和人设，用语音回她，并顺手把这件事记成家庭趣事。

这个闭环一旦成立，后面的驱虫、疫苗、喂食、猫砂盆、叫声识别、出游记录都是自然扩展。

最推荐的 MVP 顺序：

1. Home Assistant 接一个摄像头和一个模拟喂食事件。
2. Bridge 提供 `get_pet_current_state` 和 `add_pet_memory`。
3. OpenClaw 调 Bridge 工具。
4. QQ 官方机器人接入沙箱。
5. 做猫咪人格 Prompt 和家庭记忆。
6. 加 TTS 语音回复。
7. 再做 WebRTC “呼叫猫咪”页面。
